# Catty — Product Thesis

> 本文锁定 Catty 的产品本体。它取代 `agent-memory-runtime-plan.md` 里那段过时的
> "a local agent memory layer" 定位——那只是 Catty 的记忆器官，不是产品本身。
> 锁定日期：2026-05-24。

## 一句话

**一个 Claude-Code 级、开放、模型无关的通用 agent；一个实时、不断逼近你本人的副本。**
它永远预判你的下一步（让你随时知道接下来该看什么、该做什么），面对和你一样的信息做出
**比你更好**的判断，且做的始终是"你想做的事"——一个**更好的、不下班的你**。

## 它为什么不一样

它和 Claude Code / OpenClaw / Hermes 是**同一物种**（通用 agent，出厂不认识你，能力靠运行时自己长），
区别**不在"接了什么"，而在"怎么存在"**：

| | Claude Code 这类 | Catty |
|---|---|---|
| 状态 | 无状态，session 结束就忘 | **永久记得你的一切** |
| 唤起 | 你敲命令才醒 | **常驻、一直在** |
| 范围 | 一个 repo / 一次对话 | **你整台机器、整个数字生活** |
| 交互 | 终端打字 | **语音为主、桌宠为皮** |
| 主动性 | 你不问它不动 | **永远预判你下一步、主动出现、替你出手** |

与 yansu 这类"主动式 app builder"也不同：yansu 的预判**终结在"给你造一个工具"**；
Catty 的预判**终结在"你的下一步 / 我以你之名替你做"**。同一台引擎（observe→anticipate→act），
不同的魂——**yansu 是工具工厂，Catty 是另一个你**。
风险红线：如果 Catty 的"act"滑向"造工具"，就掉进 yansu 的赛道了。会话式第二自我 + 收敛成你，是护城河。

## 收敛飞轮（产品的发动机）

"不断逼近你"里那个"不断"，靠这个闭环转：

```
醒着:  观测你的活动 → 产出 self hypothesis（意图 + 下一步 + 为什么现在 + 可准备什么）
  ↑                    → judge agent 按「意图 / 动作 / 帮助性」对照现实打分（意图权重更高）
  │                                                       │ 失误 / 差距
睡着:  反省「我误解了你哪条先验」→ 写 user-prior memory / 长出 skill → dogfood 只留真让它更像你的
```

- **睡眠 / 反省 / consolidation** → 复用 `agent-memory` 的 dream 子系统。
- **自我记 memory** → 抓到自己重复犯错就写下来（本仓库 `memory/` 已在用这套）。
- **自我 build skills** → 长出自己的工具（参考 `~/mom/.agents/skills` + 全局 skills、pi 的 skill packages）。
- **dogfood** → 把新写的 memory/skill 拿到后续真实情境里跑，**只有真提升命中才留下**，否则丢弃。

这些不是几个独立 feature，是**同一个自我进化飞轮**。它们绝大多数发生在"睡眠/离线"侧，由白天的预判命中/失误喂料。

## 核心量化读数

核心量化读数是**预判命中率随时间自己往上爬（斜率为正）**——它靠自己的睡眠反省，真的吸收了你的个人信号。
但它不是单独的北极星，也不能独自证明"第二自我"已经形成。真正要守住的是：
它是否越来越像你地理解意图、判断优先级、在正确时刻准备正确的东西，并且没有用过宽预测、nudging 或提分 shortcut 污染体验。

核心赌注是：不是"能预判"，是"**会自己变得更像你地预判和判断**"。

## 自主度

**分级自主，默认"预判 + 准备 + 提议、你拍板"**，高信任的具体域随时间逐步"晋升"为自动执行。
（对应 `~/mom` 已有的"高风险先确认"习惯。）

## 架构（北极星，非 v0）

- **引擎要开放 + 模型无关**（硬约束："痛恨封闭"）。选型：
  - **pi**（earendil-works，MIT，TS，15+ provider）= 每个 actor 里的模型无关大脑，用 `streamFn` 接自有 cliproxy。
  - **Rivet Actors**（Apache-2，TS，可自托管/气隙）= actor 底座；**无执行超时 + 等审批时 hibernate 零消耗**，天然契合分级自主。
  - **agent-memory**（本仓库）= 共享的"自我"/记忆器官。
- **拓扑：一只猫在前（Catty orchestrator）+ 底层按领域裂变 sub-actor**（society of mind）。
  单一的脸保住产品的魂；底层并行的领域 actor 解掉"我只有一双眼一双手"。
- **前沿提醒**：actor 适合做 runtime/持久底座；但纯 actor 消息编排在 2026 被认为缺确定性/可观测性，
  编排逻辑该叠一层 graph/flow 纪律，而非纯消息意大利面。

## 最难的地方（核心 IP）

不是 actor、不是观测管道，而是**怎么让这堆领域专才始终是"你"，而不是一群陌生专家开会**——
"自我"（口味 / 意图 / 决策先验）必须一致地注入每个 sub-actor，否则退化成通用专家。
这就是 `agent-memory` 要扛的核心，也是整个产品的护城河和最大风险。次难点是**信任 / 自主边界**：
一个能"以你之名"出手的副本，只有当你信"它和你的差距足够小"时才敢放它做事。

## 它不是什么

- 不是 dashboard / 数据展示（那是它的脑子，不是产品）。
- 不是日记 / "今天你干了啥"的总结器（它朝前看，不朝后看）。
- 不是 app builder（不造工具，它**是你**）。
- 不绑定任何人的具体场景（mom / slock / codex 只是某一份拷贝长出来的样子，不写进产品）。
- 不是 if-else 程序：能力、取数、甚至打分，都是 agent 自己学/自己做的。

## 设计原则（来自反复踩坑）

**别用 if-else 思维设计它。** 当一个设计问题看起来需要"硬编码机制 / 客观死指标 / 场景专用集成"时，
先问：能不能让 agent 自己的智能吸收这点模糊？默认"agent 自己想办法 / judge-agent 来评"，
只给真约束留硬脚手架。产品规格停在"存在方式 / 行为"层，不下沉到 plumbing。
